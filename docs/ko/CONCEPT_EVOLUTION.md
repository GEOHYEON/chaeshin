# Chaeshin v2 — 개념 진화 정리

> 3월 26일 ~ 3월 29일 논의 전체를 시간순 + 주제별로 정리한 문서

---

## 1. 출발점: 원본 Chaeshin + OpenClaw 통합 (3/26)

### 원본 Chaeshin
- Python 기반 CBR(Case-Based Reasoning) 프레임워크
- 성공한 Tool Calling 패턴을 ToolGraph(노드+엣지 그래프)로 저장
- Retrieve → Reuse → Revise → Retain 사이클
- 불변 그래프 + 가변 실행 컨텍스트, diff 기반 리플래닝

### OpenClaw와의 관계
- OpenClaw: TypeScript/Node.js 기반 로컬 AI 비서 (멀티채널 게이트웨이)
- OpenClaw 성공 요인: Skill 시스템보다 **"내 기기에서, 내 채팅앱으로, 실제로 뭔가를 한다"는 경험**이 핵심
- 기술적으로는 로컬 게이트웨이, 멀티채널 인박스, 영속 메모리(Markdown 기반), Plugin Hook 시스템, Soul 파일, 자가 확장(새 Skill 자율 생성) 등

### 초기 통합 방향
- 단순 포팅이 아닌 **두 프로젝트의 DNA를 합쳐 재해석**
- OpenClaw = "AI that actually does things"
- Chaeshin = "AI that remembers how YOU do things"

## 2. 독립 프로젝트로 전환 (3/26)

### 핵심 결정
OpenClaw 플러그인이 아니라, **OpenClaw도 지원하는 독립 프로젝트**로 방향 전환.

### 3가지 진입점 설계

```
┌─────────────────────────────────┐
│        Core (순수 TS)           │  ← npm install 하나로
│  LifeGraph + CBR + Memory Engine│
└──────────────┬──────────────────┘
               │
    ┌──────────┴──────────┐
    │                     │
┌───▼────┐          ┌─────▼──────┐
│OpenClaw│          │  Anything  │
│ Skill  │          │  (CLI/API/ │
└────────┘          │  직접 임포트)│
                    └────────────┘
```

### CaseHub 구상
- 개발자가 코드를 올리는 게 아니라, **전문가가 노하우를 레시피처럼 공유하는 곳**
- GitHub(오픈) + App Store(검증) 이중 레이어 구조
- 도메인 특화 케이스(법률/금융/마케팅 등)가 핵심 가치
- 익명화 파이프라인이 성패를 가르는 요소

## 3. 저장 단위 논쟁 — 뺑뺑이의 시작 (3/26~3/29)

### 문제 인식: Tool Graph의 한계

**피드백 단위 불일치**
- 사용자는 **결과 단위**로 평가 ("PR 요약 좋아")
- 저장은 **실행 단위** (GitHub.getPRs → Filter → Summarize → Slack.post)
- 너무 세세해서 사람이 피드백하기 어려움

**대형 프로젝트에서 무용**
- 소규모 반복 태스크 → Tool Graph 적합
- 대형 프로젝트(아키텍처 결정, 며칠짜리 리팩토링) → Tool Graph로 표현 불가
- 중요한 건 툴이 아니라 **결정과 맥락**

**CLAUDE.md와의 경계 모호**
- CLAUDE.md = 프로젝트 맥락, 결정, 선호도
- Chaeshin Task 레이어? → CLAUDE.md랑 뭐가 다른지 불명확

### 핵심 재정의: 개발자 도구 → 도메인 전문가 플랫폼

원래 목적은 **"도메인 특화에서 유저별로 커스텀"**이었음.
- 설문조사 결과에 따라 다른 질문을 하는 구조
- 코딩하는 사람이 YAML 짜는 게 아니라
- **심리상담사/변호사/마케터**가 자기 노하우를 공유하는 것

## 4. 추상화 레벨 논쟁 (3/29)

### 뺑뺑 돈 패턴

```
구체적 Tool Graph
  ↓ "툴이 바뀌면 깨진다"
추상 플로우로 올리자
  ↓ "범용적이지 않다"
노드 타입을 정의하자
  ↓ "타입의 끝을 확신할 수 없다"
WfMC 연구 기반 9개 정의
  ↓ "그냥 그래프잖아"
줄여보자 → 2개 (do/go)
  ↓ "그것도 합칠 수 있다"
타입 없이 자연어로 가자
  ↓ "검색하려면 구조가 필요하다"
구조를 정의하자
  → 처음으로 돌아감
```

### 뺑뺑 돈 근본 원인
**두 가지 목표가 충돌**:
1. 검색/매칭을 위한 구조화 → 필드, 타입, 스키마 필요
2. 도메인 전문가의 자유로운 표현 → 자연어가 최선

하나의 저장 포맷으로 동시에 풀려고 해서 계속 충돌.

### 해결: 저장과 검색 분리

```
전문가가 자연어로 씀
       ↓
LLM이 자동으로 메타데이터 추출
       ↓
자연어는 자연어대로 저장
메타데이터는 메타데이터대로 저장
       ↓
검색: 메타데이터 + 벡터 DB
실행: LLM이 자연어 읽고 판단
```

## 5. 툴 변동성 문제와 MCP (3/29)

### 문제
Tool Graph에 구체적 API를 저장하면, 6개월 후 API 버전이 바뀌면 깨짐.

### 해결: 추상 인텐트 + MCP 런타임 바인딩

```
저장 (안 바뀜):
  survey.getScore(userId) → branch(score > 7) → notify.expert()

런타임 (바뀔 수 있음):
  survey.getScore → Typeform v2 or v3
  notify.expert  → Slack or 이메일 or 문자
```

- 비즈니스 로직은 저장, 구현 세부사항은 MCP가 결정
- **슈도코드** 개념 — 실행 가능하되 구현에 독립적

### 추상화 기준
"MCP가 결정할 수 있는 것은 노드 밖으로, 도메인 전문가가 결정해야 하는 것은 노드 안으로"
- 기본은 추상, 전문가가 원하면 구체화 가능 (CSS specificity와 유사)

## 6. 노드 타입 시스템 탐색 (3/29)

### WfMC(1993) 3축 기반 도출

| 축 | 타입 | 설명 |
|---|---|---|
| 제어흐름 | branch, parallel, loop | OR분기, AND분기, 반복 |
| 데이터 | capture, evaluate | 입력수집, 데이터판단 |
| 시간 | wait | 시간/이벤트 대기 |
| 실행 | action | 외부 시스템 호출 |
| 경계 | start, end | 시작/종료 |

### 축소 시도
- 9개 → subflow는 capture, human/event는 wait의 변형 → 결국 줄이면 do/go 2개
- **최종 판단**: 실행 주체가 LLM이면 타입 자체가 불필요할 수 있음
- 하지만 확신 못 함 → 실험 필요

## 7. 계층적 저장 구조 — Chunking 이론 (3/29)

### 인지심리학 Chunking 이론 적용
- 전문가 체스 플레이어는 개별 말이 아닌 **의미 있는 패턴 단위(chunk)**로 기억
- 전문가일수록 더 큰 chunk

### 조직 이론 Span of Control과 일치

| 수준 | 설명 | 구성 요소 수 | 근거 |
|---|---|---|---|
| 상위 (전략) | 도메인 원칙/철학 | 3~5개 | 복잡, 판단 필요 → 좁은 span |
| 중위 (패턴) | 판단 패턴/분기 | 5~7개 | 중간 복잡도 |
| 하위 (실행) | 구체적 실행 단위 | 7~15개 | 단순 반복 → 넓은 span |
| 최하위 (Tool) | 원자적 Tool Call | 1개 | MCP 바인딩 |

- Miller's Law (7±2), Chase & Simon 체스 연구(~5), Span of Control(3~8) 모두 유사 패턴

### 고정 레벨 → 트리 구조로 전환
- 레벨 수를 3~4로 고정하지 않고, **도메인 복잡도에 따라 깊이가 자연스럽게 결정**
- 복잡한 도메인 → 트리가 깊어짐
- 단순한 도메인 → 트리가 얕아짐

## 8. 불러오기(Retrieval) 전략 (3/29)

### 상위 → 하위 수직 로드
하나의 케이스를 꺼낼 때 해당 노드의 **조상(상위 맥락) + 자손(실행 단위)**을 함께 가져옴.
- 상위 레벨일수록 분류 정확도가 높음 (범위가 명확)
- 위에서 범위를 좁히고, 그 안에서 하위를 찾는 방식

### 질문 레벨 파악
- 입력 질문의 추상도에 따라 매칭 시작 레벨이 달라짐
- 벡터 유사도가 가장 높은 노드의 레벨을 자연스럽게 따라가는 방식 제안

## 9. 신경과학 접근 — Hopfield Network (3/29)

### 연상 메모리(Associative Memory) 모델
- 부분적 단서만으로 완전한 패턴 복원 (노래 몇 음 → 전곡)
- 에너지 지형: 관련 메모리의 에너지 우물이 깊어지고, 관련 없는 건 평평해짐
- **계층을 강제하지 않아도** 입력에 따라 관련 메모리들이 자연 활성화

### Hebbian Learning
- "같이 발화하는 뉴런은 연결이 강해진다"
- 자주 같이 쓰인 노드 간 연결 강화 → 자연스러운 클러스터링
- 안 쓰인 연결은 약화 → 자연스러운 Forget

## 10. 최종 방향: 계층적 분해 구조 (HTN) (3/29)

### 뇌의 목표→행동 분해 계층

```
전전두엽 (추상 목표)
  "번아웃 상담을 잘 해야 해"
  ↓
운동 전 피질 (중간 계획)
  "수면부터 확인하고 분기하자"
  ↓
일차 운동 피질 (구체적 행동)
  "지금 이 질문을 한다"
  ↓
근육 (실행)
  실제로 말함
```

### Chaeshin 적용

```
전문가가 상위 레벨로 지시
  "번아웃 고위험 유저 상담해줘"
        ↓
AI Agent가 스스로 분해
  "수면 상태 파악이 필요하다"
        ↓
더 분해
  "수면 시간을 물어봐야 한다"
        ↓
Tool Call로 변환
  ask(content="수면은 어떠세요?", tone="light")
```

### 핵심 메커니즘

```
첫 번째 실행: Agent가 처음부터 분해 (비쌈)
저장:         분해 경로를 계층 그래프로 저장
두 번째 실행: 저장된 경로 꺼내서 재사용 (저렴)
개선:         더 좋은 경로 발견하면 업데이트
```

## 11. 메모리 아키텍처 프레임워크 매핑

### Retention Process (저장)

```
Raw Data        = 에이전트의 실행 경로 로그
  ↓ Filtering
Structured Data = 분해 경로를 노드/엣지로 구조화
  ↓ Attention
Knowledge       = 계층 그래프에 저장
  ↓ Refinement (루프)
                = 성공/실패 피드백으로 지속 개선
```

### Retrieval Process (검색)

```
Memory Matching       = 벡터로 유사 노드 찾기
Neural Memory Network = 계층 탐색 (상위 맥락 + 하위 실행)
Memory Utilization    = Claude 컨텍스트에 주입 → tool calling 실행
```

## 12. claw-code (Claude Code 재구현체) 통합 포인트

### 발견 사항
- autoDream = Chaeshin의 Retain + Forget 개념과 동일
- ULTRAPLAN 모드 = 상위 노드는 강력한 모델, 하위는 가벼운 모델로 처리 가능

### 통합 구조

```
claw-code의 query_engine.py 앞단에 Chaeshin 삽입:

1. 질문 들어옴
2. Chaeshin: 질문 레벨 파악 → 그래프에서 매칭
3. 상위~하위 노드 컨텍스트 조합 → 프롬프트에 주입
4. query_engine이 enriched 컨텍스트로 LLM 호출
5. 실행 결과 → Chaeshin Retain/autoDream으로 학습
```

## 15. 한 줄 요약

**Chaeshin v2 = 도메인 전문가의 노하우를 AI가 계층적으로 분해·저장하고, 비슷한 상황이 오면 꺼내 쓰는 경험 기반 메모리 레이어. 저장은 자연어, 검색은 벡터, 실행은 MCP를 통한 Tool Calling.**
